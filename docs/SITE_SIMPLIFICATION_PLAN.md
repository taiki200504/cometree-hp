# UNION HP 簡素化・整理計画

情報量が多く「渋滞」している現状を整理し、**サイトマップで役割を明確にしたうえで**、統合・スリム化の案をまとめます。

---

## 1. 現状の問題整理

### 1.1 情報の渋滞が起きている理由

- **入口が多すぎる**: 学生向け・企業向け・About・サービス・メディア・コミュニティ・参加する・お問い合わせと、ナビだけで8本＋ドロップダウンで数十リンク。
- **1ページの重量が重い**: ひとつのページに「概要＋複数タブ＋統計＋参加形態＋CTA＋一覧」が詰まっているページが複数ある。
- **同じテーマが複数箇所に存在**: 「コミュニティ」「加盟団体」「参加方法」「学生向けメリット」などが、About・コミュニティ・参加する・for-students に分散している。
- **ハブの役割が曖昧**: 「学生向け」「コミュニティ」「参加する」が、どれも「紹介＋多数リンク」になり、**どこで何を探せばよいか**が分かりにくい。

### 1.2 「重い」ページ一覧（優先してスリム化・分割したいもの）

| ページ | 重さの内容 | 提案 |
|--------|------------|------|
| **`/community`** | 7タブ（概要・加盟団体・パートナー・ギルド・Slack・Discord・加盟団体専用）。各タブ内も説明・統計・参加形態・CTA が同居。 | **「コミュニティの入口」はタブを減らす」**。概要＋「団体一覧」「パートナー一覧」「参加する（Slack/Discord）」だけにして、細かい説明は各専門ページへリンク。 |
| **`/services`** | 学生タブ・企業タブで、それぞれ多数のカード＋説明＋CTA。 | **1ペインあたりのカード数を減らす**。代表的な3〜5個に絞り、残りは「詳細は○○」リンクか別ページへ。 |
| **`/join`** | 参加方法（学生/企業）＋FAQ＋連携コミュニティ一覧＋CTA。 | **「参加の入口」に一本化し、詳細はサブページへ**。FAQ・連携一覧は別ページ or 折りたたみにして、トップは「誰向けに何ができるか」だけにする。 |
| **`/about`** | PMVV・カルチャー・事業リンク・統計など。 | **スクロール量を減らす**。核心メッセージ＋「詳しくは事業内容・メンバーへ」に絞る。 |
| **`/community/slack`** | チャンネル一覧・機能説明・参加ボタンなど。 | **「Slackに参加する」＋短い説明＋リンク**にスリム化。詳細は Slack 側または1ページに集約。 |
| **`/community/guild`** | ギルドの説明が長い。 | **「ギルドとは」を短く＋「詳しくは○○」**にしてもよい。 |
| **`/`（トップ）** | ニュース・統計・サービス案内・ポッドキャスト・CTA 等。 | **役割を「初めての人に何を伝えるか」に絞る**。ニュース数本＋最大1〜2個のCTA＋メディア/コミュニティへの導線に整理。 |

## 2. 重複・分散している情報（統合候補）

| テーマ | 今ある場所 | 統合の方向性 |
|--------|------------|--------------|
| **「学生には何があるか」** | `/for-students`, `/for-students/benefits`, `/services?tab=student`, `/about/union`, `/about/students` | **学生向けは「for-students ＝ 目次＋短い説明」1枚にし、詳細は benefits / union / services などへリンク**。about/students は about か for-students に統合または削除検討。 |
| **「団体には何があるか」** | `/for-organizations/benefits`, `/community`（タブ内）, `/join/organization` | **団体向けは「登録団体のメリット」1本＋加盟申請への導線**に整理。community 概要では「団体一覧・パートナー一覧」へのリンクに留める。 |
| **「企業には何があるか」** | `/services?tab=corporate`, `/join/corporate`, `/contact` | **企業向けは「サービス紹介（スリム）」＋「提携・協賛は join/corporate」「問合せは contact」**に役割を分離。同じ説明を3ページに書かない。 |
| **「コミュニティの全体像」** | `/community`（7タブ）, `/about/community` | **「コミュニティ」は1ページで全体像を短文＋リンクに。about/community は about に統合または廃止**。 |

## 3. 整理後のサイト構造（案）

### 3.1 ナビの見直し（シンプル化）

- **学生向け**: 「学生向け」1項目 → クリックで **1ページ**（for-students：短文＋「ユニメイト・学生団体連合・イベント・拠点・メリット・サービス」へのリンク）。ドロップダウンをやめて1ページで完結させる案も可。
- **企業向け**: 「企業向け」→ サービス（スリム）・提携/協賛（join/corporate）・お問い合わせの3本に整理（現状維持だが中身を軽く）。
- **About**: 理念・メンバー・事業・支援者・スタッフ募集 に整理。about/community は廃止または about に統合。
- **サービス**: 1ページのみ（学生/企業タブは残してよいが、カード数・文量を半減）。
- **メディア**: メディア概要・ポッドキャスト・学生ニュース・掲示板。番組個別はポッドキャスト配下で十分。
- **コミュニティ**: **「コミュニティ」1ページを軽く** → 加盟団体一覧・提携企業一覧・Slack・Discord・団体専用ポータル へのリンク中心。タブを減らし、概要・参加形態は短文＋リンクに。
- **参加する**: 「参加のご案内」1ページ（誰向けに何があるかの選択肢＋各ページへリンク）。団体・企業・スタッフ・寄付はサブとして維持。
- **お知らせ**: 現状どおり。
- **お問い合わせ**: 現状どおり（フォーム・寄付・サービス申込へのリンク）。

### 3.2 ページごとの「重さ」の目安（整理後）

- **入口・ハブ**: 見出し＋短い説明＋**リンク5〜8本程度**。長文・複数タブ・大量カードは持たない。
- **詳細ページ**: 1テーマ1ページ。スクロールはあるが「このページは○○の説明」と役割がはっきりしていること。
- **一覧ページ**: 一覧＋絞り込み or ページネーション。余計な説明は最小限。

## 4. 実行の優先順位（案）

1. **サイトマップの確定**  
   - 上記を踏まえ、**本当に残すページ**と**統合・廃止するページ**を一覧で確定する（`SITEMAP.md` に「整理後」列を追加してもよい）。

2. **「重い」ページのスリム化（即効きく）**  
   - `/community`: タブ削減・概要は短文＋リンク中心。  
   - `/services`: 学生・企業それぞれカード数と文言を削る。  
   - `/join`: 参加方法の概要だけにして、FAQ・連携一覧は別ページ or 折りたたみ。

3. **重複コンテンツの統合**  
   - about/community → about に統合 or 廃止。  
   - about/students → for-students または about に統合。  
   - 学生向け・団体向け・企業向けの「メリット」「紹介」が複数にある場合、**代表1ページを決め、他はリンクに置き換え**。

4. **ナビ・フッターの整理**  
   - リンク数を減らし、「学生向け」「コミュニティ」「参加する」のドロップダウンを短くする。  
   - 全ページ一覧（ALL）は、整理後のサイトマップと同期する。

5. **トップページの整理**  
   - 伝えたいメッセージを1〜2個に絞り、ニュース・イベント・メディア・参加への導線をシンプルに。

---

## 5. 次のステップ

- **意思決定**: 上記の「廃止・統合」「スリム化の方針」を採用するか、一部だけ適用するかを決める。
- **整理後サイトマップ**: 確定した構成で `SITEMAP.md` に「現状」と「整理後」の2列を用意し、差分からタスクリストを作る。
- **実装**: 重いページのスリム化 → 統合 → ナビ整理 → トップ の順で進めると、見た目の変化が分かりやすい。

必要なら、特定のページ（例: `/community` や `/services`）だけを対象に、**具体的な文言案・削るブロックのリスト**も別ドキュメントで起こせます。
